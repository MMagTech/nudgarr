Throttling

i
How many search commands are sent to your indexer at once per batch. A batch size of 1 means each item is searched individually with a pause in between. Higher values send multiple searches simultaneously which can overwhelm indexers and trigger rate limiting. Keep this at 1 unless you have a specific reason to increase it.
Number of items sent per search command. Smaller values are easier on your indexers.
i
How long Nudgarr pauses between each batch of searches. Combined with batch size this controls the overall pace of a sweep. A sleep of 5 seconds with a batch size of 1 means one search every 5 seconds. Lowering this too much reduces the breathing room between searches and increases the risk of hitting indexer rate limits.
Pause between batches in seconds. Gives your indexers time to breathe.
i
Adds a random delay on top of the sleep time between batches. If sleep is 5 seconds and jitter is 2 seconds, actual pauses will vary between 5 and 7 seconds. This randomness makes Nudgarr's search pattern less predictable and helps avoid triggering automated rate limit detection that looks for suspiciously regular request intervals.
Random extra pause on top of Sleep Seconds to help avoid indexer rate limiting

⚠️ INDEXER RATE LIMITS

Nudgarr instructs instances to search on your behalf. Start with low limits and a sensible cooldown. Indexers may rate limit or ban if hit too hard. Please be respectful of their limits.

Backlog Nudges

Searches movies and episodes identified as missing. These searches are on top of your Max Per Run caps.

Radarr

i
How many missing movies are searched per Radarr instance each sweep when backlog nudges are enabled. Unlike Cutoff Unmet searches which target items you already have at a lower quality, missing searches target items you have never downloaded. These can add up fast — if you have hundreds of missing movies and set this high combined with a frequent cron schedule you can generate a very large number of searches in a short time. Start at 1 and increase slowly.

What's New in v3.1.0

Here's what changed since your last visit.

🗄️ SQLite Database
All history, stats, and exclusions now live in a local SQLite database. Existing data migrates automatically on first start — no action required.
🕐 Cron Scheduling
The run interval is now a cron expression — default 0 */6 * * *. Set a TZ env var to match your timezone. Nudgarr no longer runs on startup — the first sweep fires when the cron next fires or you press Run Now. Existing intervals are converted automatically on upgrade.
⏭️ Skip Queued
Items already in your download queue are silently skipped during sweeps. They never consume a search slot — your max per run is always filled from items that actually need a nudge.
🔄 Import Iteration Tracking
The Imports tab now tracks how many times an item has been through the full search-and-download cycle. Re-imports show a ×2, ×3 badge, and turnaround measures from the very first search.
🔗 Open in Radarr / Sonarr
Click any title in History or Imports to jump straight to that item in Radarr or Sonarr. Hover to see which app will open.

No Instances Configured

Add a Radarr or Sonarr instance to get started. The Sweep tab will populate automatically once an instance is added.

Per-Instance Overrides — How it works

Read this once — it won't show again.

🌐 Global values are the starting point
Override any field on any card independently. Unset fields always fall back to global.
⚠️ Search volume multiplies per instance
Each instance runs independently. 4 instances × max 15 = up to 60 searches per sweep. Check your indexer before raising any per-instance caps.
🎯 Nothing changes until you hit Apply
Enabling this tab doesn't change any behaviour. Overrides only take effect when you explicitly Apply them on a card.
🔄 Turning this off reverts all instances to global immediately, cooldowns included
Override values are preserved and restored if you re-enable.
📋 Each instance can have backlog on or off
The global backlog toggle no longer controls everything when overrides are active.
